## Objectif

I show here a simple and concrete example: using an API during Docusaurus build, saving the result in a static JSON file, then displaying this data in a page.

This type of flow is ideal for keeping a static site in production while leveraging external data at compile time.

## Workflow GitHub Actions

Pour les builds CI/CD, nous utilisons GitHub Actions pour récupérer les données de l'API lors du déploiement. Cette approche est plus fiable et gère les échecs d'API de manière élégante.

Pour le développement local, un script Node.js (`scripts/return-api-data.js`) peut être utilisé pour pré-générer le fichier JSON statique, simulant ainsi l'environnement de production.

The workflow `.github/workflows/main.yml` includes an API call step that:

1.  **Tente d'appeler l'API principale** (Sling Academy) avec des en-têtes de navigateur réalistes.
2.  **Valide la réponse** pour s'assurer qu'il s'agit bien d'un JSON valide avec un statut de succès.
3.  **Utilise une API de secours (Lorem Picsum)** si l'API principale échoue ou renvoie des données invalides.
4.  **Utilise un fichier de secours local** si l'API de secours échoue également.

<Snippet filename=".github/workflows/main.yml" source=".github/workflows/main.yml" />

Cette approche robuste garantit que le build réussit toujours, même si les API externes sont temporairement indisponibles.

## Afficher les Photos sur une Page

La page `src/pages/photos.jsx` lit le JSON statique et affiche les photos.

<Snippet filename="src/pages/photos.jsx" source="src/pages/photos.jsx" />

La page charge `/json/photos-data.json`. En mode développement (lancé via `npm start`), si ce fichier n'existe pas, la page se rabat sur un chargement direct depuis l'API pour afficher les images.

Voir la page en action /photos/

## Créer le Composant `PhotosGrid`

Le composant `src/components/PhotosGrid/index.js` est l'élément qui rend la grille de photos dans votre article.

It does several important things:

-   il tente d'abord de charger le fichier statique `/json/photos-data.json`
-   si ce fichier n'existe pas en développement, il se rabat automatiquement sur l'API Sling Academy (notre API de démonstration)
-   il gère l'état de chargement (`loading`), les erreurs et le cas où aucune photo n'est disponible
-   il limite le nombre de photos affichées avec la prop `limit`
-   il construit une grille réactive avec les cartes Docusaurus

<Snippet filename="src/components/PhotosGrid/index.js" source="src/components/PhotosGrid/index.js" />

The component is exposed in MDX via `src/theme/MDXComponents.js`, which allows calling it directly in an article with:

```html
<PhotosGrid limit={6} />
```

This creation is a practical option: it allows cleanly inserting data into an MDX page without duplicating the fetch and display logic.

## Why It's Interesting

- you preload data at build time
- you avoid making API requests on the client side in production
- you can deploy to a static host without a backend server

## Simple Alternative

If you want, you can replace the React page with an MDX page, but the principle remains the same:

1. fetch the API before the build
2. save data in `static/`
3. consume this static file in your page

> This article shows how to use an API at build time with a light and easy-to-reproduce example.
