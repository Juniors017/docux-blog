## Goal

I show here a simple and concrete example: using an API during Docusaurus build, saving the result in a static JSON file, then displaying this data in a page.

This type of flow is ideal for keeping a static site in production while leveraging external data at compile time.

## GitHub Actions Workflow

Instead of using a Node.js script during local build, we use GitHub Actions to fetch API data during deployment. This approach is more reliable for CI/CD environments and handles API failures gracefully.

The workflow `.github/workflows/main.yml` includes an API call step that:

1. **Tries the primary API** (Sling Academy) with realistic browser headers
2. **Validates the response** to ensure it's proper JSON with success status
3. **Falls back to Lorem Picsum** if the primary API fails or returns invalid data

<Snippet filename=".github/workflows/main.yml" source=".github/workflows/main.yml" />

This robust approach ensures the build always succeeds, even if external APIs are temporarily unavailable.

See the page in action [/photos/](/photos/)

## Create the `PhotosGrid` Component

The component `src/components/PhotosGrid/index.jsx` is the element that renders the photo grid in your article.

It does several important things:

- it first tries to load the static file `/json/photos-data.json`
- if this file doesn't exist in development, it automatically falls back to the Sling Academy API (our demo API)
- it manages loading state (`loading`), errors, and the case where no photos are available
- it limits the number of displayed photos with the `limit` prop
- it builds a responsive grid with Docusaurus cards

<Snippet filename="src/components/PhotosGrid/index.js" source="src/components/PhotosGrid/index.js" />

The component is exposed in MDX via `src/theme/MDXComponents.js`, which allows calling it directly in an article with:

```html
<PhotosGrid limit={6} />
```

This creation is a practical option: it allows cleanly inserting data into an MDX page without duplicating the fetch and display logic.

## Why It's Interesting

- you preload data at build time
- you avoid making API requests on the client side in production
- you can deploy to a static host without a backend server

## Simple Alternative

If you want, you can replace the React page with an MDX page, but the principle remains the same:

1. fetch the API before the build
2. save data in `static/`
3. consume this static file in your page
